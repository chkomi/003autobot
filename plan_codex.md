# 003 Autobot Codex Plan

## 1. 목표

이 프로젝트의 목표는 OKX 거래소 데이터를 기반으로 다음 기능을 제공하는 웹 플랫폼을 구축하는 것이다.

- 종목별 롱 투자 가능성 점수
- 종목별 숏 투자 가능성 점수
- 백테스팅 실행 및 결과 비교
- 일간/주간/월간 총괄 투자 리포트
- 전략 평가, 점검, 운영 관리
- 투자 성과 평가 및 추적

핵심 방향은 기존 Python 트레이딩 코어를 최대한 재사용하고, Vercel에는 프론트엔드를 올리며, 장시간 실행이 필요한 실시간 수집과 백테스트는 별도 Python 런타임으로 분리하는 것이다.

## 2. 현재 상태

현재 레포에는 이미 다음 자산이 준비되어 있다.

- OKX REST/WebSocket 연동 코드
- 전략 점수 계산 로직
- 멀티 심볼 백테스트 코드
- FastAPI 기반 기존 대시보드
- SQLite 기반 저장 구조
- Vercel 배포용 `web/` Next.js 초기 골격
- 아키텍처 및 MVP 로드맵 문서

즉, 완전 신설 프로젝트가 아니라 기존 엔진을 서비스 구조로 재편하는 단계라고 보는 것이 맞다.

## 3. 권장 아키텍처

### 3.1 프론트엔드

- 위치: `web/`
- 배포: Vercel
- 기술: Next.js App Router + TypeScript
- 역할:
  - 종목 랭킹 대시보드
  - 종목 상세 분석 페이지
  - 백테스트 요청/결과 조회 페이지
  - 총괄 리포트 페이지
  - 관리자용 상태 점검 페이지

### 3.2 Python API 서비스

- 역할:
  - 현재 점수 조회
  - 종목별 세부 분석 조회
  - 백테스트 작업 생성/상태 조회
  - 리포트 데이터 제공
  - 운영 상태 점검 데이터 제공

- 권장 엔드포인트:
  - `GET /api/v1/market/leaderboard`
  - `GET /api/v1/market/symbols/{symbol}`
  - `GET /api/v1/market/symbols/{symbol}/score`
  - `POST /api/v1/backtests`
  - `GET /api/v1/backtests/{jobId}`
  - `GET /api/v1/reports/overview`
  - `GET /api/v1/ops/health`

### 3.3 Worker 서비스

- 역할:
  - OKX 실시간 시세 수집
  - 스냅샷/랭킹 캐시 갱신
  - 주기적 점수 재계산
  - 장시간 백테스트 실행
  - 리포트 생성 자동화

### 3.4 데이터 저장소

- PostgreSQL:
  - 종목 메타데이터
  - 점수 이력
  - 백테스트 작업 및 결과 메타데이터
  - 포트폴리오/성과 리포트
- Redis:
  - 실시간 점수 캐시
  - 리더보드 캐시
  - 작업 진행률 상태
  - 실시간 스트리밍용 임시 데이터
- Blob/S3 계열 스토리지:
  - CSV 리포트
  - PDF/HTML 리포트
  - 대용량 백테스트 산출물

## 4. 왜 이렇게 나누는가

Vercel은 프론트엔드와 짧은 요청 처리에는 강하지만, 지속 연결형 WebSocket 수집기나 장시간 백테스트 워커 운영에는 적합하지 않다. 따라서 아래처럼 역할을 분리하는 것이 가장 안전하다.

- Vercel = 사용자 웹 화면
- Python API = 데이터 조회와 작업 제어
- Python Worker = 실시간 수집과 무거운 계산

이 구조가 운영 안정성, 배포 단순성, 성능 면에서 가장 현실적이다.

## 5. 제공할 핵심 기능

### 5.1 종목별 롱/숏 점수

- 전 종목 대상 실시간 랭킹
- 롱 점수, 숏 점수, 중립도 표시
- 점수 근거:
  - 추세
  - 모멘텀
  - 거래량
  - 변동성
  - 심리/매크로 요소
- 종목 상세 페이지에서 점수 산출 근거 설명

### 5.2 백테스팅

- 단일 종목 백테스트
- 기간 선택 백테스트
- 전략 버전별 결과 저장
- 종목 간 비교 랭킹
- 핵심 지표:
  - 총 수익률
  - 연환산 수익률
  - 승률
  - Profit Factor
  - Sharpe Ratio
  - MDD
  - 거래 횟수
  - 평균 보유 시간

### 5.3 총괄 리포트

- 일간 리포트
- 주간 리포트
- 월간 리포트
- 포함 내용:
  - 전체 수익 요약
  - 종목별 기여도
  - 잘된 전략/부진한 전략
  - 리스크 상태
  - 다음 액션 제안

### 5.4 평가/점검/관리

- 데이터 수집 정상 여부
- 점수 계산 최신성
- 백테스트 최신성
- 리스크 한도 초과 여부
- 포지션 노출 현황
- API 상태/지연/실패 감시

### 5.5 투자 성과 평가

- 전략별 성과 비교
- 종목별 성과 비교
- 실거래 대비 백테스트 괴리 분석
- 벤치마크 대비 초과 수익 분석
- 성과 추세 추적

## 6. 재사용 우선 대상

현재 코드베이스에서 우선 재사용할 대상은 아래와 같다.

- `strategy/score_engine.py`
- `dashboard/server.py`
- `run_full_backtest.py`
- `backtest/`
- `database/`
- `data/okx_rest_client.py`
- `data/okx_ws_client.py`

즉, 새 기능의 핵심은 기존 로직을 버리는 것이 아니라 API와 서비스 계층으로 감싸는 것이다.

## 7. 단계별 실행 계획

## Phase 1. 기반 정리

목표:

- 서비스 구조를 분리하고 개발 기준을 정리한다.

작업:

1. GitHub 레포 연결 완료
2. Vercel용 `web/` 초기 구조 완료
3. 문서 정리 완료
4. 다음 단계로 Python API 구조 설계

완료 기준:

- 레포 푸시 가능
- 웹 앱 기본 빌드 성공
- 개발 경로 문서화 완료

## Phase 2. API 계층 만들기

목표:

- 기존 Python 코어를 웹에서 호출 가능한 API로 바꾼다.

작업:

1. 점수 조회 API 구현
2. 리더보드 API 구현
3. 종목 상세 API 구현
4. 백테스트 작업 생성 API 구현
5. 백테스트 상태 조회 API 구현
6. 운영 상태 점검 API 구현

완료 기준:

- 프론트엔드가 mock data 없이 API 응답으로 화면을 렌더링할 수 있음

## Phase 3. 데이터 저장소 전환

목표:

- SQLite 중심 구조에서 서비스형 저장 구조로 전환한다.

작업:

1. PostgreSQL 스키마 설계
2. 심볼/점수/리포트/백테스트 작업 테이블 설계
3. Redis 캐시 구조 설계
4. 기존 SQLite 의존 흐름 정리

완료 기준:

- 실시간 데이터와 이력 데이터를 분리해 저장 가능

## Phase 4. Worker와 스케줄링

목표:

- 실시간 수집과 무거운 계산을 분리 운영한다.

작업:

1. OKX WebSocket 수집 워커 구성
2. 점수 스냅샷 갱신 작업 구성
3. 백테스트 비동기 작업 큐 구성
4. 일간/주간 리포트 생성 작업 구성

완료 기준:

- 웹 요청과 무거운 계산이 분리되어 안정적으로 동작

## Phase 5. 웹 UI 연결

목표:

- Vercel 프론트엔드에 실제 데이터를 연결한다.

작업:

1. 리더보드 페이지 연결
2. 종목 상세 페이지 연결
3. 백테스트 요청 폼 연결
4. 백테스트 결과 테이블 연결
5. 리포트 페이지 연결
6. 운영 점검 페이지 연결

완료 기준:

- 실제 API 데이터로 운영 가능한 MVP 화면 완성

## Phase 6. 운영 기능 강화

목표:

- 투자 운영 도구로서의 완성도를 높인다.

작업:

1. 알림 시스템 연동
2. 장애/지연 감지
3. 리스크 경보
4. 전략 버전 비교
5. 성과 평가 체계 정교화

완료 기준:

- 일일 운영과 리뷰를 이 시스템 안에서 수행 가능

## 8. MVP 범위

첫 배포에서 꼭 필요한 기능은 아래로 제한한다.

- 실시간 종목 랭킹
- 종목별 롱/숏 점수
- 종목 상세 분석
- 단일 종목 백테스트 생성/조회
- 일간 총괄 리포트
- 운영 상태 점검 화면

이 범위를 넘는 기능은 2차 단계에서 붙인다.

## 13. 판단 기준

앞으로 구현 중 의사결정이 필요할 때는 아래 기준을 따른다.

- 기존 Python 코어 재사용을 우선한다.
- Vercel은 프론트엔드 중심으로 쓴다.
- 장시간 작업은 서버리스에 억지로 넣지 않는다.
- MVP를 먼저 만들고 이후 확장한다.
- 실시간성보다 운영 안정성을 우선한다.
