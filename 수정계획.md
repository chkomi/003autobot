## 1단계: 긴급 수정 (오늘 내 완료)

### [긴급-1] Kelly 계산 버그 — `risk_manager.py` L137~138

#### 문제점

```python
# risk_manager.py L127~138 (현재 코드)
stats = await self._db.fetch_trade_stats()
...
if stats.get("total", 0) >= 10:
    total = stats["total"]
    wins = stats.get("wins", 0) or 0
    win_rate = wins / total if total > 0 else 0.5
    avg_win_r = abs(stats.get("avg_win", 0.02) or 0.02) / signal.entry_price   # ← 버그
    avg_loss_r = abs(stats.get("avg_loss", 0.01) or 0.01) / signal.entry_price  # ← 버그
```

`fetch_trade_stats()`가 반환하는 `avg_win`과 `avg_loss`는 DB의 `pnl_usdt` 컬럼 평균값으로, **절댓값 USDT 금액**이다 (예: +50달러, -30달러). 이것을 `signal.entry_price`(예: BTC $50,000)로 나누면:

```
avg_win_r = 50 / 50,000 = 0.001 → 0.1%
```

이 수치는 "BTC 1개 가격 대비 수익률"이라는 **무의미한 값**이다. Kelly Criterion이 필요로 하는 것은 **투자 원금 대비 수익률(R-배수)**이다. 예를 들어 원금 $500을 투자해 $50을 벌었다면 `avg_win_r = 0.10` (10%)이어야 한다.

또한, `position_sizer.py`의 `_kelly_fraction` 함수에서 이 비율을 사용해 `full_kelly`를 계산하므로, 잘못된 입력값은 Kelly 분수 전체를 왜곡한다.

#### 수정 방법

DB에서 `pnl_pct` (이미 `trades` 테이블에 존재하는 컬럼) 기반 평균을 가져오도록 수정한다.

**Step 1 — `db_manager.py` `fetch_trade_stats()` 수정 (L263~264)**

```python
# database/db_manager.py — fetch_trade_stats() 내 SQL 수정
# 기존:
AVG(CASE WHEN pnl_usdt > 0 THEN pnl_usdt END) AS avg_win,
AVG(CASE WHEN pnl_usdt <= 0 THEN pnl_usdt END) AS avg_loss,

# 수정 후:
AVG(CASE WHEN pnl_usdt > 0 THEN pnl_usdt END) AS avg_win,
AVG(CASE WHEN pnl_usdt <= 0 THEN pnl_usdt END) AS avg_loss,
AVG(CASE WHEN pnl_pct > 0 THEN pnl_pct END) AS avg_win_pct,    # 추가
AVG(CASE WHEN pnl_pct <= 0 THEN pnl_pct END) AS avg_loss_pct,  # 추가
```

**Step 2 — `risk_manager.py` L137~138 수정**

```python
# risk_manager.py L137~138 수정 후
avg_win_r = abs(stats.get("avg_win_pct", 0.02) or 0.02)   # pnl_pct 기반 (이미 비율)
avg_loss_r = abs(stats.get("avg_loss_pct", 0.01) or 0.01) # pnl_pct 기반 (이미 비율)
```

`pnl_pct` 컬럼은 `trade_lifecycle.py`에서 청산 시 계산하여 저장되므로, 값이 없는 경우(초기 운영, 거래 수 < 10회)는 기존 기본값(0.02, 0.01)이 그대로 사용된다.

#### 예상 효과

- Kelly 계산이 실제 투자 원금 대비 수익률을 반영하게 됨
- 포지션 크기의 과대/과소 산정 오류 제거
- 거래 이력이 쌓일수록 Kelly 분수가 의미 있는 값으로 수렴

### [긴급-2] 포지션 사이저 하드코딩 버그 — `position_sizer.py` L91~92

#### 문제점

```python
# position_sizer.py L91~92 (현재 코드)
quantity = round(notional / entry_price, 4)  # ← 소수점 4자리 하드코딩 (BTC 전용)
quantity = max(quantity, 0.001)              # ← 최소 0.001 하드코딩 (BTC 전용)
```

OKX의 심볼별 최소 주문 단위와 수량 정밀도는 자산마다 다르다.

| 심볼 | 가격(예시) | 최소 주문 | OKX 정밀도 | 현재 코드 결과 | 올바른 값 |
|------|-----------|-----------|-----------|--------------|---------|
| BTC | $50,000 | 0.001 BTC | 소수 3자리 | 정상 | 0.001 |
| SUI | $2 | 1 SUI | 정수 | `max(250.0000, 0.001)` → 정상 보임 | 1 |
| GMT | $0.20 | 10 GMT | 정수 | `max(2500.0000, 0.001)` → 정상 보임 | 10 |

수량 자체는 우연히 맞을 수 있으나, **최솟값 0.001** 보정 로직이 모든 알트코인에 적용되면 최악의 경우 SUI $2짜리를 0.001개($0.002)만 주문하는 오류가 발생한다. 또한 `L99~103` 로그에 "BTC 수량" 단위로 출력되어 디버깅도 혼란스럽다.

```python
# position_sizer.py L155~159 — 로그도 BTC 하드코딩
logger.info(
    f"리스크 검증 PASS: {signal.direction} | "
    f"수량: {pos_size.quantity:.4f} BTC | "   # ← "BTC" 하드코딩
    f"증거금: ${pos_size.margin_required:.2f}"
)
```

def calculate(

    balance_usdt: float,
    entry_price: float,
    stop_price: float,
    symbol: str = "BTC-USDT-SWAP",   # 추가

## 2단계: 중요 수정 (오늘~내일)

### [중요-1] 신호 빈도 위기 — 삼중 필터 완화

#### 문제점

신호가 0건 발생하는 원인은 세 개의 필터가 **AND 조건**으로 겹쳐 있기 때문이다.

```
[필터1] regime.block_ranging = true
        → ADX < 20.0이면 진입 차단
        → signal_aggregator.py L129: is_ranging AND block_ranging → FLAT 반환

[필터2] signal_combine_mode = "AND"
        → Layer2(MACD 상향크로스 최근 3봉 내 + RSI 범위) AND
          Layer3(BB%B <= 0.35 또는 >= 0.65 + 거래량 ≥ 평균×1.2)
        → 두 조건이 동시에 충족되는 빈도 매우 낮음
        → signal_aggregator.py L172: layers_ok = layer2_pass and layer3_pass

[필터3] scoring.weak_signal_threshold = 65
        → 위 두 레이어를 통과해도 스코어가 65점 미만이면 FLAT
        → signal_aggregator.py L260: score_result.total < sp.weak_signal_threshold → 진입 거부
```

각 필터의 개별 통과율을 추정하면:
- 횡보 차단: 전체 시간의 ~40~60%가 ADX < 20 구간
- AND 모드: Layer2 통과율 ~15% × Layer3 통과율 ~20% = 동시 통과 ~3%
- 스코어 65점 이상: 위 3%를 통과한 시그널 중 추가 ~50%만 통과

누적 통과율: **60% × 3% × 50% ≈ 0.9%** → 60분 주기 신호 체크 시 약 111시간에 1번

#### 수정 방법

세 필터를 각각 단계적으로 완화한다.

| 최대 드로다운 | ≤ 25% |

15건 미만이면 [개선-1]의 추가 완화 파라미터를 적용 후 재실행.

### 4. 시뮬레이션 드라이런 (선택)

실제 OKX 연결 전 `--dry-run` 모드로 24시간 운영하여 신호 발생 빈도와 포지션 사이즈를 로그로 확인:

```bash
python main.py --dry-run --log-level DEBUG 2>&1 | tee dry_run.log
grep "리스크 검증 PASS\|시그널 진입\|FLAT" dry_run.log
```
