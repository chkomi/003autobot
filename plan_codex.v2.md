## 1. 목표

이 프로젝트의 목표는 OKX 거래소 데이터를 기반으로 다음 기능을 제공하는 웹 플랫폼을 구축하는 것이다.

- 종목별 롱 투자 가능성 점수
- 종목별 숏 투자 가능성 점수
- 백테스팅 실행 및 결과 비교
- 일간/주간/월간 총괄 투자 리포트
- 전략 평가, 점검, 운영 관리
- 투자 성과 평가 및 추적

핵심 방향은 기존 Python 트레이딩 코어를 최대한 재사용하고, Vercel에는 프론트엔드를 올리며, 장시간 실행이 필요한 실시간 수집과 백테스트는 별도 Python 런타임으로 분리하는 것이다.

## 2. 현재 상태

현재 레포에는 이미 다음 자산이 준비되어 있다.

- OKX REST/WebSocket 연동 코드
- 전략 점수 계산 로직
- 멀티 심볼 백테스트 코드
- FastAPI 기반 기존 대시보드
- SQLite 기반 저장 구조
- Vercel 배포용 `web/` Next.js 초기 골격
- 아키텍처 및 MVP 로드맵 문서

즉, 완전 신설 프로젝트가 아니라 기존 엔진을 서비스 구조로 재편하는 단계라고 보는 것이 맞다.

## 3. 권장 아키텍처

### 3.1 프론트엔드

- 위치: `web/`
- 배포: Vercel
- 기술: Next.js App Router + TypeScript + React Query(SWR 대체 가능)
- 역할:
  - 종목 랭킹 대시보드
  - 종목 상세 분석 페이지
  - 백테스트 요청/결과 조회 페이지
  - 총괄 리포트 페이지
  - 관리자용 상태 점검 페이지

### 3.2 Python API 서비스

- 프레임워크: FastAPI + Uvicorn/Gunicorn
- 호스팅 후보: Fly.io / Render / Railway 중 1택 (초기엔 Fly.io 권장 — 리전 고정·WS 친화)
- 역할:
  - 현재 점수 조회
  - 종목별 세부 분석 조회
  - 백테스트 작업 생성/상태 조회
  - 리포트 데이터 제공
  - 운영 상태 점검 데이터 제공

- 권장 엔드포인트:
  - `GET /api/v1/market/leaderboard`
  - `GET /api/v1/market/symbols/{symbol}`
  - `GET /api/v1/market/symbols/{symbol}/score`
  - `POST /api/v1/backtests`
  - `GET /api/v1/backtests/{jobId}`
  - `GET /api/v1/reports/overview`
  - `GET /api/v1/ops/health`

### 3.3 Worker 서비스

- 런타임: Python 프로세스 (API와 동일 호스팅 내 별도 서비스)
- 큐: RQ (Redis 기반, 단순성 우선). 규모 커지면 Dramatiq/Celery로 승격
- 스케줄러: APScheduler 또는 호스팅의 cron 기능
- 역할:
  - OKX 실시간 시세 수집 (WS)
  - 스냅샷/랭킹 캐시 갱신 (초~분 주기)
  - 주기적 점수 재계산
  - 장시간 백테스트 실행
  - 리포트 생성 자동화

### 3.4 데이터 저장소

- PostgreSQL:
  - 종목 메타데이터
  - 점수 이력
  - 백테스트 작업 및 결과 메타데이터
  - 포트폴리오/성과 리포트
- Redis:
  - 실시간 점수 캐시
  - 리더보드 캐시
  - 작업 진행률 상태
  - 실시간 스트리밍용 임시 데이터
- Blob/S3 계열 스토리지 (Cloudflare R2 / AWS S3):
  - CSV 리포트
  - PDF/HTML 리포트
  - 대용량 백테스트 산출물

## 4. 왜 이렇게 나누는가

Vercel은 프론트엔드와 짧은 요청 처리에는 강하지만, 지속 연결형 WebSocket 수집기나 장시간 백테스트 워커 운영에는 적합하지 않다. 따라서 아래처럼 역할을 분리하는 것이 가장 안전하다.

- Vercel = 사용자 웹 화면
- Python API = 데이터 조회와 작업 제어
- Python Worker = 실시간 수집과 무거운 계산

이 구조가 운영 안정성, 배포 단순성, 성능 면에서 가장 현실적이다.

## 5. 제공할 핵심 기능

### 5.1 종목별 롱/숏 점수

- 전 종목 대상 실시간 랭킹
- 롱 점수, 숏 점수, 중립도 표시
- 점수 근거:
  - 추세
  - 모멘텀
  - 거래량
  - 변동성
  - 심리/매크로 요소
- 종목 상세 페이지에서 점수 산출 근거 설명
- 점수 갱신 주기: 리더보드 캐시 30~60초, 상세 계산 1~5분 (MVP 기준)

### 5.2 백테스팅

- 단일 종목 백테스트
- 기간 선택 백테스트
- 전략 버전별 결과 저장
- 종목 간 비교 랭킹
- 핵심 지표:
  - 총 수익률
  - 연환산 수익률
  - 승률
  - Profit Factor
  - Sharpe Ratio
  - MDD
  - 거래 횟수
  - 평균 보유 시간

### 5.3 총괄 리포트

- 일간 리포트
- 주간 리포트
- 월간 리포트
- 포함 내용:
  - 전체 수익 요약
  - 종목별 기여도
  - 잘된 전략/부진한 전략
  - 리스크 상태
  - 다음 액션 제안

### 5.4 평가/점검/관리

- 데이터 수집 정상 여부
- 점수 계산 최신성
- 백테스트 최신성
- 리스크 한도 초과 여부
- 포지션 노출 현황
- API 상태/지연/실패 감시

### 5.5 투자 성과 평가

- 전략별 성과 비교
- 종목별 성과 비교
- 실거래 대비 백테스트 괴리 분석
- 벤치마크 대비 초과 수익 분석
- 성과 추세 추적

## 11. 보안 및 인증 (신규)

- OKX API Key / Secret은 Worker 런타임 환경변수로만 보관. 레포·클라이언트 노출 금지.
- 관리자 페이지는 단일 사용자 기준 Bearer Token 또는 BasicAuth로 MVP 수준 보호.
- 공개 엔드포인트는 rate limit(요청/분) 적용.
- Next.js → API 호출은 서버 컴포넌트/Route Handler에서 API 키를 주입 (브라우저 노출 금지).
- 시크릿 회전 정책: 3개월 주기 수동, 유출 의심 시 즉시.

## 12. 관측성 (신규)

- 로그: JSON 구조 로그. 수집기/큐 작업은 job_id 기준 추적 가능해야 함.
- 메트릭:
  - API: RPS, p50/p95 레이턴시, 5xx 비율
  - Worker: 큐 지연, 실패율, 재시도 횟수
  - WS 수집기: 메시지/초, 재연결 횟수, 시퀀스 갭
- 알림:
  - WS disconnect 5분 지속 → 텔레그램 경보
  - 점수 스냅샷 10분 이상 지연 → 경보
  - 백테스트 작업 실패율 10% 초과 → 경보
- 에러 트래킹: Sentry (API·Worker·Web 공통).

## 13. 테스트 전략 (신규)

- 단위 테스트: pytest. 점수 엔진·백테스트 핵심 함수는 커버리지 80% 목표.
- 골든 스냅샷: 특정 심볼·기간에 대한 점수/백테스트 결과를 고정값으로 저장 → 회귀 방지.
- 통합 테스트: API + Redis + Postgres를 docker-compose로 띄워 end-to-end 호출.
- 프론트엔드: 주요 페이지 Playwright smoke test.
- CI: GitHub Actions. PR마다 lint + unit + goldens 실행.

## 14. 로컬 개발 & 배포 (신규)

- 로컬: `docker-compose.yml`에 Postgres + Redis + API + Worker 정의. Web은 `pnpm dev`로 별도 실행.
- 환경 파일: `.env.local`(개발) / `.env.staging` / `.env.production` 분리. 커밋 금지.
- 배포:
  - Web: Vercel (main push 자동)
  - API/Worker: GitHub Actions → Fly.io deploy
- 환경: dev → staging → prod 3단계. staging은 실시간 데이터 수집 on, 실제 주문은 항상 off.

## 15. 데이터 이관 (신규)

- 기존 SQLite 데이터는 일회성 ETL 스크립트로 PostgreSQL에 적재.
- 전환 기간(약 1~2주) 동안 이중 쓰기(dual-write) 유지, 결과 비교로 정합성 확인.
- 컷오버 후 SQLite는 7일 보관 후 아카이브.

## 16. 시간 및 기준값 표준 (신규)

- 저장: 모든 타임스탬프 UTC (`TIMESTAMPTZ`).
- 표시: 웹 UI 기본 KST, 사용자 토글 지원.
- 캔들 경계: OKX 표준에 맞춤 (UTC 기준 open/close).
- 리포트 기준일: KST 00:00 기준의 전일/전주/전월.

## 18. 리스크와 대응

### 리스크 1. Vercel에 실시간 수집과 장시간 작업을 모두 올리려는 시도

대응: 수집기와 백테스트는 별도 Python 런타임으로 분리.

### 리스크 2. 기존 SQLite 구조가 확장성 한계에 걸릴 가능성

대응: PostgreSQL 중심으로 빠르게 이관.

### 리스크 3. 실시간 데이터와 백테스트 데이터 모델이 섞이는 문제

대응: 실시간 캐시와 이력 저장 구조를 분리.

### 리스크 4. 점수 산식 신뢰도 검증 부족

대응:
- 점수 근거 표시
- 전략 버전 관리
- 백테스트와 실거래 성과 비교 기능 추가

### 리스크 5. OKX rate limit / 차단 (신규)

대응:
- 클라이언트에 요청 토큰버킷 적용
- 심볼별 폴링 주기 차등화
- 장애 시 지수 백오프 + jitter

### 리스크 6. WebSocket 단절·시퀀스 갭 (신규)

대응:
- 자동 재연결 + 하트비트 감시
- 마지막 수신 시퀀스 체크포인트 → 복구 시 REST로 공백 보강
- disconnect율 메트릭 경보

### 리스크 7. 캐시-DB 정합성 불일치 (신규)

대응:
- 캐시 키에 스냅샷 버전 포함
- 기준 진실은 항상 DB, 캐시는 TTL 있는 사본
- 재계산 시 invalidate-then-write 순서 고정

## 20. 판단 기준

앞으로 구현 중 의사결정이 필요할 때는 아래 기준을 따른다.

- 기존 Python 코어 재사용을 우선한다.
- Vercel은 프론트엔드 중심으로 쓴다.
- 장시간 작업은 서버리스에 억지로 넣지 않는다.
- MVP를 먼저 만들고 이후 확장한다.
- 실시간성보다 운영 안정성을 우선한다.
- 기술 선택은 "가장 단순한 것 → 한계가 보이면 승격" 순서로 한다.
